# Ukrainian STT WER corpus

Labelled audio used by `training/evaluation/stt_wer.py` to measure **WER / CER**
of the production STT backend on real Ukrainian smart-home commands (thesis
§4.3.4). Without this corpus the WER figure cannot be measured — `stt_wer.py`
reports `measured: false` and refuses to invent a number.

## Files

| File | Tracked in git? | Purpose |
|------|-----------------|---------|
| `commands_uk.txt` | ✅ yes | Reference transcripts, one per line (the WER ground truth). |
| `record.py` | ✅ yes | Interactive recorder → writes WAV clips + `manifest.jsonl`. |
| `manifest.jsonl` | ❌ no (gitignored) | `{"audio", "text"}` per line — generated by `record.py`. |
| `clip_*.wav` | ❌ no (gitignored) | Your recorded audio. Voice recordings stay on the device. |

Audio + manifest are gitignored on purpose: voice recordings are personal data
and should not leave the recording machine.

## Where to record — RPi 5 vs MacBook

Two metrics come out of this corpus, and they have different hardware
sensitivities:

* **WER / CER** depends on the *STT engine* + the *acoustic capture*, **not** on
  the CPU. Recording on a MacBook is fine — the engine (faster-whisper-base
  int8, `uk`) is identical. The only caveat: a laptop near-field mic in a quiet
  room is cleaner than the RPi 5 far-field mic in a real room, so MacBook WER is
  *slightly optimistic*. Acceptable for the thesis **if** §4.3.4 notes the audio
  was captured on a laptop mic and real-environment WER is left to the usability
  study (already listed in §4.4.2).
* **Latency** (STT p95, e2e ≤5 s for NFR-2) is **hardware-bound** — a MacBook
  run is ~an order of magnitude too fast and must **not** back the NFR-2 claim.

**Recommended:** record the clips wherever is convenient (MacBook mic is fine),
then `scp` the `clip_*.wav` + `manifest.jsonl` to the RPi 5 and run the scoring
(`stt_wer.py`, `voice_e2e_latency.py`) **there**. That gives a representative WER
*and* a representative latency, with only the acoustic capture being laptop-clean.
If you score on the MacBook too, use the WER (with the clean-mic disclaimer) but
label the latency as dev-hardware and don't cite it for NFR-2.

## How to record

```bash
# one-time deps (RPi also needs the PortAudio system lib)
sudo apt install libportaudio2
uv pip install sounddevice soundfile

# record — speak each prompted line; ENTER to start/stop, s=skip, q=quit
uv run python -m training.evaluation.stt_corpus.record
```

Re-running resumes after the clips already in `manifest.jsonl`, so you can
record in several sittings. ~30–50 clips is enough for a stable WER on this
closed command set.

## How to score

```bash
# uses the same backend as production (STT_BACKEND / FASTER_WHISPER_MODEL env)
uv run python -m training.evaluation.stt_wer

# or the whole suite
make evaluate-stt
```

Output: `materials/evaluation_results/stt_wer.{json,md}` with corpus WER, CER,
and p50/p95 latency. Set `STT_BACKEND=hailo` to score the opt-in NPU Whisper
instead of the CPU faster-whisper engine on the same clips.

## Editing the command list

Keep transcripts spelled exactly as you expect the STT to output them — they are
the WER references. The list already mixes clean commands with a "dirty" block
(colloquial forms, русизми) that probes robustness on realistic input.
